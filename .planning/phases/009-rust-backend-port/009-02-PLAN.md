**Function: pub fn build_quadegg_from_connectivity(connectivity: &Array2<i32>, n_verts: usize) -> Array2<i32>**

Algorithm (2-phase construction from Python reference):

**Phase 1: Create edge list**
1. Iterate each element in connectivity
2. For each element edge (v0, v1):
   - Triangles (3 vertices): edges are (0,1), (1,2), (2,0)
   - Quads (4 vertices): edges are (0,1), (1,2), (2,3), (3,0)
3. Store edges as (min_vertex, max_vertex) canonical form in HashMap<(i32, i32), edge_id>
4. Count unique edges; allocate edges: Array2<i32>::zeros((n_edges, 4))
5. For each unique edge, assign edge_id and initialize:
   - edges[edge_id] = [v0, v1, -1, -1] (opposite_edge, next_edge unassigned)

**Phase 2: Pair opposites + assign next**
1. For each element, iterate edges in order (0→1→2→... per element type)
2. For each edge (v0, v1):
   - Look up opposite edge (v1, v0) in edge map
   - If found: set edges[edge_id, 2] = opposite_edge_id
   - If not found (boundary): set edges[edge_id, 2] = -1
3. For each edge within same element, assign next edge in rotation:
   - Next edge within element forms a cycle (e0→e1→e2→e0 for triangle, etc.)
   - Set edges[edge_id, 3] = next_edge_id

**Validation (per RESEARCH.md Pitfall 1):**
- Every edge has opposite_id or -1 (no None)
- If opposite_id != -1, then edges[opposite_id, 2] = edge_id (reciprocal)
- All next_edge_ids form valid cycles per element

Return edges: Array2<i32>

Rationale: Quad-edge is 2-phase O(n) construction. Phase 1 creates edge list with unique edges. Phase 2 pairs opposites and assigns next. Sentinel value -1 indicates boundary (no opposite edge). Mixed-element support: triangles and quads handled uniformly via element vertex count.

Implement 6 converter functions in adjacency.rs, returning Array2<i32> or HashMap for Python marshalling:

1. **to_edge2vert(edges: &Array2<i32>) -> Array2<i32>**
   - Extract unique edges in canonical (sorted) form: [min(v0,v1), max(v0,v1)]
   - Return Array2 [n_edges, 2], sorted by vertex IDs (matching Python reference)

2. **to_elem2edge(edges: &Array2<i32>, connectivity: &Array2<i32>, n_elems: usize) -> Array2<i32>**
   - For each element, find incident edge IDs from edges array
   - Return Array2 [n_elems, 3] or [n_elems, 4] padded (triangles: 3 cols, quads: 4 cols)

3. **to_vert2edge(edges: &Array2<i32>, n_verts: usize) -> Vec<Vec<usize>>**
   - For each vertex, collect incident edge IDs
   - Return Vec<Vec<usize>> (List[List[int]] in Python); convert to PyList via PyO3

4. **to_vert2elem(connectivity: &Array2<i32>, n_verts: usize) -> Vec<Vec<usize>>**
   - For each vertex, collect incident element IDs
   - Return Vec<Vec<usize>>

5. **to_edge2elem(edges: &Array2<i32>) -> Array2<i32>**
   - Extract edge-to-element adjacency from edges array (opposite_id, opposite_id for each edge)
   - Handle boundary edges (opposite_id = -1)
   - Return Array2 [n_edges, 2] with -1 sentinels for boundary

6. **to_elem2vert(connectivity: &Array2<i32>) -> Array2<i32>**
   - Return connectivity array as-is (already in element-to-vertex form)

All converters return types marshallable to Python via PyO3 (Array2<i32> → PyArray2<i32>, Vec<Vec<usize>> → PyList).

Rationale: Six converters provide all common adjacency queries. Canonical form (sorted vertex pairs) ensures bit-identical output vs Python reference. PyO3 marshalling handles conversion automatically via Py type system.
