Implement `src/chilmesh_core/io.rs` module with two public functions:

**Function 1: parse_fort14(path: &str) -> Result<RustMesh, RustMeshError>**

Logic:
1. Read file to string using std::fs::read_to_string(path) — wrap std::io::Error in RustMeshError::IOError
2. Split into lines; strip whitespace
3. Parse header (line 0): "NE NP [NBND]"
   - NE = number of elements (u32)
   - NP = number of vertices (u32)
   - NBND = optional number of boundary nodes (skip if present)
   - Validate: NE > 0 && NP > 0; else ParseError
4. Parse element connectivity (lines 1 to NE):
   - Format: "IE ITYPE V1 V2 [V3 V4]"
   - IE = element ID (1-indexed, skip/validate)
   - ITYPE = element type (3=triangle, 4=quad)
   - Vertices: V1, V2, ... are 1-indexed in file; convert to 0-indexed for Rust
   - Store in connectivity: Array2<i32>::zeros((NE, 4))
     - Triangles: [v0, v1, v2, v0] (padded to 4, per RESEARCH.md Pitfall 2)
     - Quads: [v0, v1, v2, v3]
   - Store element type in elem_type: Vec<u32>
   - Validate: all vertex IDs in [0, NP); else VertexOutOfBounds error
5. Parse coordinates (lines NE+1 to NE+NP):
   - Format: "X Y [BND_FLAG]"
   - X, Y = f64 coordinates
   - Skip optional BND flag (boundary node marker)
   - Store in points: Array2<f64>::zeros((NP, 2))
   - Validate: all coordinates finite (!NaN, !Inf); else InvalidGeometry error
6. Return RustMesh { points, connectivity, elem_type, n_verts: NP, n_elems: NE }

**Function 2: write_fort14(mesh: &RustMesh, path: &str) -> Result<(), RustMeshError>**

Logic:
1. Build output string with header: format "NE NP" where NE = mesh.n_elems, NP = mesh.n_verts
2. Write elements (rows of connectivity):
   - For each row i in connectivity:
     - Determine ITYPE from elem_type[i]: 3 for triangle, 4 for quad
     - Format: "{} {} {} {} {} {}" with 1-indexed vertex IDs
     - Include fourth vertex only if ITYPE == 4
3. Write coordinates:
   - For each row in points: format "{:.8e} {:.8e}" (8-digit precision per SPEC.md)
4. Write to file using std::fs::write(path, output_string); wrap errors in RustMeshError::IOError
5. Return Ok(())

Rationale: Fort.14 is ADCIRC ASCII format (old mesh standard). Parser must handle 1-indexed file format, mixed element types (3 vs 4), padding triangles per RESEARCH.md Pitfall 2, and boundary markers per Pitfall 3. Use Result<T, RustMeshError> for error propagation per CONTEXT.md decision. Error variants: ParseError (format/value), VertexOutOfBounds (ID range), InvalidGeometry (NaN/Inf), IOError (file access).
  </action>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Rust ↔ Python (PyO3 FFI) | Untrusted input from Python (file paths, coordinates, vertex IDs) crosses into Rust |
| Fort.14 file format | External ASCII file with potential malformed content (tabs, invalid integers, NaN coordinates) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-009-01 | Tampering | Fort.14 header (NE, NP integers) | Mitigate | Validate NE, NP parse as valid u32; bounds-check all vertex IDs in [0, NP). Invalid header → ParseError → Python OSError. |
| T-009-02 | Tampering | Coordinate data (f64 values) | Mitigate | Check f64::is_finite() for all coordinates; reject NaN/Inf with InvalidGeometry error. |
| T-009-03 | Information Disclosure | RustMesh struct internals | Mitigate | RustMesh is opaque to Python (#[pyclass], no public fields). All access via methods; no memory layout leakage. |
| T-009-04 | Denial of Service | Huge mesh allocation | Accept | No mitigation in Phase 009; document memory requirements. If mesh requires >1 GB RAM, expect OOM. Out-of-scope for library code. |
| T-009-SC | Supply Chain | Cargo dependencies (pyo3, ndarray, thiserror) | Mitigate | All packages from crates.io with [VERIFIED] status in RESEARCH.md Package Legitimacy Audit. No [SUS] or [SLOP] packages. Cargo.lock pins versions. |
